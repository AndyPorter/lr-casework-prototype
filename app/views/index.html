The Service

Digital Mortgage is the service. How do we process, monitor and administer the internal elements of the service.

Internal Processing

What might a User interface for staff look like. How these digital mortgages could be/are going to be registered. What 'digital' work looks like.

Based upon -

  • what we've learnt about mortgage processing/casework from staff and stakeholders
  • what we've learnt about (and from) our people (staff)

Design Principles

General design principles live in the design manual. Specific principles and guidance not yet general enough are listed here.

Beta(from November 2015)

'phase 1' (in team delivery plan)

Very linear flow, mirroring current process, no case/application is fully automatic - every case has multiple manual steps

Main block to automation - prior application check (daylist). Unblock daylist and UI flow likely to be very different/efficient

Happy Flow = 1) pick from case list -> 2) application has priority? -> 3) auto drafted register ok? -> 4) complete prior applications (mark-off) -> 5) complete mortgage application (mark-off)

Alternate flows: from steps 2, 3, and 4 in happy flow = 1) capture failure reasons 2) process in current casework systems (send case to ERSV).

Happy Flow, Steps 2 and 4 require person to check, and take action, in existing systems

Design constraints / parameters / assumptions

Given the expectation of very few applications.. what is the minimum thing we can do, and then iterate rapidly (based on adding most value)...

  • tens of applications a day (less than 100)
  • only 1 or 2 staff using system (so they will share same list - no need to 'lock' applications
  • no need to login - application/system logs will record who did what
  • settings staff need (such as friendly name - not staff id) will be hardcoded (no need for UI to manage settings)
  • staff based physically close to development team (preferably embeded)
  • staff in contact with each other
  • staff empowered/have enough knowledge to accept draft registers without guidance from this system
  • development team will support - primary contact point for problems, and will manage errors/support
  • case progress is not stored - user must complete steps 1 to 5, or start again
  • both staff could start to process the same case - but first to end will complete (mark it off) - no case locking - satff to manage work between them - not the system
  • when case sent to ESRV - other staff will process* (*we wont be able to monitor full application journey)
  • not designed for all staff - may not be fully accessible
  • designed for desktop device and modern browser - lastest + latest-1 version of firefox/chrome/IE
  • the system does not contain instructions on how to use itself (system should not need a manual)
  • on day 1 - the system will only contain minimal practice/processing instructions
  • the system will resolutely make minimal integration with current systems - only where absolutely necessary to interact with a register or manage applications
  • the system should give the staff using it feedback on progress/cases completed
  • minimum required = a list, a flow to process, a flow to capture problems, feedback to staff on progress, basic metrics, trust in the system, close working with dev team (one team)
  • will need re designing if changes to the above occur
