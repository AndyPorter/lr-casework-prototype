Processing digital mortgages - DesignUI for staff, service thinking, user exerience.

What might a User interface for staff look like. How these digital mortgages could be/are going to be registered. What 'digital' work looks like.

Based upon -

  • what we've learnt about mortgage processing/casework from staff and stakeholders
  • what we've learnt about (and from) our people (staff)
  • what other government departments are doing - how they administer and internally process services
  • what does digital world / work look like (like services you would choose to use at home)

Design Principles

General design principles live in the design manual. Specific principles and guidance not yet general enough are listed here.

User interfaces - phase 2Now... (from November 2015)

Digital mortgages to be routed and some/all processing automated. Either routed to exisiting estate (e.g. CORA), or routed to 'new world'. New world to be designed to process as much as possible automatically. Some steps might be easier/preferable to get a user to do...

morgtage application -> has priority? -> drafted register -> check register change.

  • has priority - needs human check? if human - check in a) exisitng systems, b) new ui
  • check register change - needs human check? if human - a) check whole register?, b) a part of the register? [i.e. how much is auto checked and how much needs human]

The Service

Digital Mortgage is the service, most/all to be automated. How do we monitor and administer the internal elements of the service.

  • some sort of performance / admin ui
  • ability to look at 'cases' / applications
  • task -> ability to see the mortgage application has priority - application ahead in the queue is a the paceholder for this mortgage (towel on the deckchair) (OS(W) - search)
  • task -> approve/fail a draft register / section of a register / other name for a part of the whole. Or many different small tasks - e.g. check restriction, check name discrepancy, mismatch address, etc - TBD
